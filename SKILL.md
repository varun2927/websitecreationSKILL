# Website & App Discovery

A structured discovery interview that extracts everything needed before a single line of code
is written. Covers purpose, audience, pages, features, design direction, content, technical
requirements, performance, SEO, and launch readiness.

**Always run this before building.** A website built without discovery is a website rebuilt
from scratch after the first feedback round.

---

## How to Run This Skill

Work through the phases **conversationally** — don't dump all questions at once. Ask a
section, get answers, then move to the next. If the user gives rich answers that cover
multiple sections, acknowledge and skip ahead. Adapt to their familiarity level.

At the end, produce a **Discovery Summary Document** — a clean, structured brief that can
be handed to any developer or designer to start building immediately.

---

## Phase 1 — Purpose & Problem

*Understand the "why" before anything else.*

Ask:
1. **What is this website/app for?** What problem does it solve, or what does it help
   someone do? (One clear sentence is the goal.)
2. **Who is it for?** Describe the person who will use it — their age, tech comfort level,
   what they're trying to accomplish when they land on the site.
3. **What is the single most important thing a visitor should do on this site?**
   (Buy something? Sign up? Contact you? Read content? Book an appointment?)
4. **Does a similar website already exist?** Share 2–3 examples — sites you like, or
   competitors. Note: what do you like about them and what would you do differently?
5. **Why does this need to exist now?** What's the context — new business, rebrand,
   replacing an old site, launching a product, building an audience?

---

## Phase 2 — Pages & Structure

*Map the full site before designing a single screen.*

Ask:
1. **What pages does the site need?** (Homepage, About, Services, Pricing, Blog, Contact,
   Login, Dashboard, etc.) Let them list freely, then suggest obvious missing ones.
2. **Is there a logged-in/logged-out experience?** Does the site have user accounts? If yes:
   - What can a logged-out visitor see vs a logged-in user?
   - What does a user's "account" or "dashboard" contain?
3. **Is there an admin panel?** Does someone need to manage content, users, orders,
   or data from a back-end interface?
4. **What is the navigation structure?** How should pages link together — simple top nav,
   sidebar, mega-menu, mobile hamburger? Walk through the main user journey.
5. **Are there any hidden or protected pages?** Password-protected content, invite-only
   sections, premium tiers?

---

## Phase 3 — Features & Functionality

*Every feature is a scope decision. Get them all on the table.*

Walk through this checklist with the user — ask which apply and get details on each:

### Content & Media
- [ ] Blog or articles (who writes, how often, categories/tags needed?)
- [ ] Image gallery or portfolio
- [ ] Video embeds or hosted video
- [ ] Downloadable files (PDFs, resources, lead magnets)
- [ ] Search functionality

### User Actions
- [ ] Contact form (what fields? where does it send?)
- [ ] Newsletter / email signup (which platform — Mailchimp, ConvertKit, etc.?)
- [ ] User registration and login
- [ ] User profiles (what info do users have/edit?)
- [ ] Commenting or reviews

### Commerce & Payments
- [ ] Products or services for sale (physical, digital, or subscription?)
- [ ] Shopping cart and checkout
- [ ] Payment processing (Stripe, Razorpay, PayPal?)
- [ ] Invoicing or order management
- [ ] Coupons or discount codes

### Bookings & Scheduling
- [ ] Appointment booking (Calendly integration or custom?)
- [ ] Event listings or ticketing
- [ ] Availability/calendar management

### Integrations
- [ ] CRM (HubSpot, Salesforce, Zoho?)
- [ ] Analytics (Google Analytics, Mixpanel, Hotjar?)
- [ ] Live chat (Intercom, Crisp, Tawk.to?)
- [ ] Social media feeds or sharing
- [ ] Third-party APIs (maps, weather, stock data, etc.)

### Dynamic / App Features
- [ ] Real-time updates (notifications, live data, websockets?)
- [ ] AI features (chatbot, recommendations, generation?)
- [ ] Multi-language / localization
- [ ] Dark mode toggle
- [ ] Progressive Web App (installable on mobile?)

For each checked item — ask: *What exactly should it do? Any specific behavior or edge case?*

## Phase 4 — Look & Feel

*Design direction without a single mockup.*

Ask:
1. **What words describe the feeling the site should give?**
   (e.g., professional, playful, minimal, bold, trustworthy, luxurious, techy, warm, clean)
2. **Color direction:** Any colors that must be used (brand colors)? Any that must be
   avoided? Preference for light mode, dark mode, or both?
3. **Typography feel:** Should text feel editorial and refined, clean and modern, bold and
   expressive, or friendly and approachable?
4. **Layout style preference:** Lots of whitespace and minimalism, or dense and
   information-rich? Card-based? Grid-based? Single long scroll?
5. **Reference sites for look & feel:** Share URLs of sites whose *visual style* you
   admire — even unrelated industries.
6. **Logo and brand assets:** Does a logo exist? Brand guidelines? Icon set?
   If not, does one need to be created?
7. **Photography/illustration style:** Real photos, illustrations, icons only, abstract
   backgrounds, or AI-generated imagery? Is existing photography available, or does it
   need to be sourced?
8. **Motion and animation:** Should the site feel static and fast, or should it have
   scroll animations, hover effects, transitions? Any inspiration for interaction style?

---

## Phase 5 — Content

*Content gaps kill launches. Find them now.*

Ask:
1. **Who is writing the copy?** The user themselves, a copywriter, or should the AI
   draft it based on the brief?
2. **What content already exists?** Existing website copy, product descriptions, About page
   text, team bios, blog posts?
3. **What content needs to be created from scratch?** List each page and its content status:
   *exists / needs updating / needs creating.*
4. **Are there legal pages needed?** Privacy policy, Terms of service, Cookie consent,
   Refund policy, Disclaimer?
5. **What media assets exist?** High-res logo files, product photos, team photos, brand
   video, background images?
6. **Is there a tone of voice guide?** How formal/informal should the writing be? Any
   words or phrases to always use or avoid?

---

## Phase 6 — Technical Requirements

*Stack, hosting, and constraints.*

Ask:
1. **Does a tech stack preference exist?** (React, Next.js, Vue, plain HTML, WordPress,
   Webflow, Shopify, etc.) If they don't know — ask what matters: full control, no-code
   ease, specific integrations, or maximum performance?
2. **Where will this be hosted?** (Vercel, Netlify, AWS, shared hosting, self-hosted,
   no preference?)
3. **Is there an existing domain?** Or does one need to be registered? What is it?
4. **What existing systems must this integrate with?** (Current CRM, email platform,
   payment system, internal tools?)
5. **Are there compliance requirements?** GDPR (EU users?), CCPA (California users?),
   accessibility (WCAG AA required?), industry-specific regulations?
6. **Who maintains the site after launch?** A developer, or does the client need to update
   content themselves? (Determines CMS complexity needs.)
7. **Is there an existing codebase to work with, or is this greenfield?**

---

## Phase 7 — Performance & SEO

*Speed and discoverability baked in from the start.*

Ask:
1. **How important is Google ranking for this site?** Is organic search a key traffic
   source, or is this primarily accessed via direct link/social/ads?
2. **Are there target keywords?** What would the ideal customer type into Google to find
   this? List 5–10 phrases.
3. **What is the target load time?** Should the site feel instant (< 1s), fast (< 2s),
   or is performance not a priority?
4. **Who is the geographic audience?** Local city, national, or global? (Affects CDN,
   language, and local SEO strategy.)
5. **Is there existing domain authority?** Old site with backlinks? Domain age?
6. **Are there Core Web Vitals requirements?** (LCP, CLS, FID — usually matters for
   e-commerce and high-traffic content sites.)
7. **Will there be structured data needs?** (Product schema, review schema, event schema,
   FAQ schema — needed for rich Google results.)

---

## Phase 8 — Launch & Beyond

*Define done and what comes after.*

Ask:
1. **What is the launch deadline?** Hard date, soft target, or "as soon as possible"?
2. **What does "done" look like?** What must be working for launch, and what can come
   post-launch?
3. **What is the budget range?** (Helps recommend tech stack and scope — no-code vs
   custom code, premium plugins vs custom builds.)
4. **Who is on the team?** Solo founder, in-house developer, hired freelancer, agency?
   Who approves work?
5. **How will success be measured?** Traffic, signups, sales, time-on-site, ranking
   position, form submissions? What does a win look like at 3 months post-launch?
6. **What is the plan for ongoing content?** Blog cadence, social posting, product updates?
   (Determines whether a CMS is essential or optional.)
7. **Is a staging / preview environment needed?** Should changes be reviewed before going
   live?

## Anti-Patterns to Avoid

- **Never start building before Phase 1–3 are complete.** Even if the user says "just
  start coding" — get the minimum: purpose, audience, pages, and one design reference.
- **Never assume the tech stack.** Ask first. A non-technical founder may need Webflow;
  a developer may want Next.js. They're not the same build.
- **Never skip the content phase.** "We'll add content later" is how sites launch with
  placeholder text. Surface the gap now.
- **Never treat a vague answer as enough.** "Something modern" is not a design direction.
  Push for references, examples, or at least 3 descriptive words.
- **Never let scope creep hide in "maybe later."** Log it in the brief under
  "Post-launch scope" so it's acknowledged but bounded.
